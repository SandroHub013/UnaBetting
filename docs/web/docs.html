Overview

UnaBetting is an open-source tennis analytics project with two halves: a leak-free machine-learning pipeline (ATP/WTA) and a desktop app that orchestrates it. The project's defining choice is honesty — it publishes the real out-of-sample numbers, including the ones that prove it has no betting edge, instead of hiding behind a cherry-picked metric.

No betting edge. The model reaches ~67% accuracy out-of-sample, but the honest backtest loses money to the bookmaker margin. UnaBetting is a research and discipline tool, not betting advice. 18+, gamble responsibly.

Quick start

# clone & install
git clone https://github.com/SandroHub013/UnaBetting.git
cd UnaBetting && pip install -r requirements.txt
cp .env.example .env   # add your API keys

# build the data & models
python -m src.data.download
python -m src.data.clean
python -m src.features.build_features
python -m src.models.train
python -m src.models.backtest   # honest backtest

# launch the app (native window on Windows)
python -m src.dashboard

The in-app chat (UnaBettingOS) needs Ollama with a tool-calling model (default qwen3.5:9b).

Architecture

Data flows from public sources through a strictly temporal feature pipeline into calibrated models, then into live inference and the app's cockpit.

src/
  data/        download · clean · odds scraper (book allowlist)
  features/    ELO · player stats · clutch · build_features
  models/      train (anti-leak) · honest backtest · cross-validation
  betting/     signals (value vs sharp + CLV) · portfolio (bet tracker)
  live/        live inference · news agent · web research
  dashboard/   the app (FastAPI + pywebview + xterm.js)

The app runs a local FastAPI server bound to 127.0.0.1 and opens a native window. REST endpoints serve the cockpit; WebSockets drive the pipeline runner, the real terminals, and the chat agent.

The honest result

Every headline number is produced under three non-negotiable rules — temporal split, randomized perspective, train-only medians. Here is where things actually stand:

MetricValue
Model accuracy (ensemble, test 2025+)67.4%
Odds-ensemble (real-odds rows)69.6%
Log loss0.601
ROC AUC0.740
Honest backtest ROI-7.3%
Backtest win rate48.5%

Backtest run 2026-06-12: 2,801 matches, 458 bets, max drawdown 43.8%. Decent accuracy, but the honest backtest still loses to the bookmaker margin — there is no betting edge. We publish this on purpose: it's the difference between research and marketing.

The leak hunt

This project's real history is a chain of self-caught data leaks. Each one made the model look brilliant; each one was a bug.

2026-05-21 — Pre-split imputation

Medians computed over the full dataset leaked the future distribution into training. acc 78.8% → fixed with train-only medians.

2026-05-22 — Unpaired perspective columns

A winner-side feature with no loser twin survived perspective randomization and let the model reconstruct the target through the 50% flip. ROI +46% → fixed by enforcing perspective pairs.

2026-06-09 — The fake 85% backtest

No scaler, un-randomized rows, and a 0.5/0.5 implied-prob sentinel that became phantom even-money odds. The bankroll "exploded" — on a bug. win rate 85% → −92% honest ROI.

CLV — the metric of truth

Win/loss on single bets is noise. Closing Line Value — whether the price you took beats the closing sharp line — is the only provable edge. UnaBetting snapshots multi-book odds on a schedule, builds a no-vig sharp consensus (Pinnacle/Betfair), and measures CLV per signal. Sell nothing until CLV is positive and stable.

Data cockpit

Bold color-block dashboards over your own betanalytix.db: bankroll, signals, bets, the model trajectory and the edge distribution — all sortable and filterable, in six themes and five languages.

UnaBettingOS

An agentic memory core: a local LLM (Ollama, default qwen3.5:9b) wired through whitelisted tools to your live data, the Obsidian vault (search_knowledge), the knowledge graph (query_graph) and a persistent, git-versioned memory (save_memory / recall_memory). It can also browse the web. Everything stays on your machine.

3D knowledge graph

Self-evolving loops

Following the "write loops, not prompts" philosophy, scheduled headless agents run the project — each matched to a model by difficulty:

  • PR review (every 30 min) — reviews and merges contributors' PRs after tests, secret scans and leak-free checks.
  • Code review (periodic) — deep review of models and system.
  • Nightly maintenance — data refresh, conditional retrain, metrics, honest backtest.
  • Results check — compares live-scanned predictions to real Sofascore outcomes.
  • Docs/repo sync — keeps docs, the live graph and the public repo in step.

Contributing

Pick an item from the backlog, branch, make one focused change, prove it leak-free, open a PR with before/after numbers. The PR-review loop takes it from there. Full rules in CONTRIBUTING.md.

The golden rule: every accuracy claim must be proven leak-free. Otherwise it's a bug, not a result.
